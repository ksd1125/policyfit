✦ 정책핏(PolicyFit) AI 챗봇 — 구현 인수인계서

작성 2026-06-03 · 세션 N · 다음 세션에서 그대로 착수 가능 · 프로젝트: mail-handoff-lite-20260601-1745

한 줄 요약. 디자인 목업 정책핏 (오프라인).html을 기준으로 "검색이 아닌 진단" 컨셉의 정책 챗봇을 신규 구축한다. 데이터는 기존 정제·분류 py 체인을 재사용한 자동 갱신 파이프라인으로 안정 수급한다. 전문가/관리자 모드 포함 5종 유형 전체.

1. 디자인 사양 (목업 디코드 결과)

브랜드·톤

  • 이름 정책핏, 슬로건 "복잡한 공고문은 그만. 대화 한 번으로 내 정책을 찾으세요."
  • 컬러 토스 블루 #3182F6, 폰트 Pretendard, 별(✦) 로고, 모바일 앱 느낌
  • 핵심 가치 "검색이 아닌 진단" — 몇 가지 질문 → 맞춤 사업 자동 선별

5종 사용자 유형 (온보딩 "어떤 분이세요?")

① 소상공인·자영업자(운영 중) ② 예비창업자 ③ 정책 담당자 ④ 일반 시민 ⑤ 상담원

유형이 흐름을 바꿈: 자동 채움 / 질문 생략 / 관점 변경 (예: 운영 중 → 창업자금 제외, 예비창업자 → 창업 관점)

진단 질문 (유형별 분기, "1개만 답해도 결과")

  1. 목적(최우선): 운영자금·대출 / 매출·판로 / 디지털 전환 / 창업 자금·공간 / 인건비·고용
  2. 사업 단계: 창업 준비 중 / N년 미만 / N년 이상
  3. 업종(선택): 음식·외식 / 도소매·유통 / 생활·서비스 / 온라인·플랫폼 / 제조·기술
  4. 지역(선택): 경기·인천 / 광역시 / 그 외

선택지 칩에 건수 표기("N건 보기"), "건너뛰고 전체 보기", "이 질문은 건너뛰었어요". 디자인 의도: 괄호 숫자 = 최종 결과 건수와 일치.

결과 (매칭도 순)

  • "조건에 맞는 지원사업 N건을 매칭도 순으로 정리했어요"
  • 핵심 카드: 목적 · 지원금액 · 기간 · D-day · 매칭도% · 자격 · 준비물 · 신청절차
  • 0건: "비슷한 사업·알림 보기"(막다른 길 금지). 면책 + 소상공인 통합콜센터 1357

부가 기능

  • 조건 저장 / 새 공고 알림(카톡·문자 — 프로토는 로컬)
  • 전체 둘러보기: 유형별 분류 카드(분야별 모집 중 사업 수)
  • "왜 정책핏인가요?": 기존 포털(키워드·수백건·PDF) vs 정책핏(진단·핵심카드·용어풀이)
  • 전문가/관리자 모드: 정책 입안자 브리핑(차트), 관리자 상담 현황 — "분야별 1건당 평균 문의", "마감 임박 N건", "문의 집중 Top3", "공고 신설·예산 확대 검토" 등

데이터 스키마(디자인 기대 필드)

id, title, org, category, tags[], summary, purpose,
amountLabel, amountValue, amountSub, period, dday, match,
eligible[], prepare[], steps[], status(모집중/마감/예산소진),
goals[], stages[], regions[], note

2. 데이터 수급 (안정성) — 핵심 답변

Q. 자동 수급하면 정제·분류도 되나?
A. 예. 새 공고를 수집한 뒤 기존 py 체인을 그대로 재실행하면 목적/금액/대상/서류/섹션 분류와 중복 제거가 동일하게 적용된다. 정책핏의 purpose·amount·match·prepare·steps는 이 산출물에서 매핑된다.

2-1. 재사용 체인 (이미 완성)

collect-bizinfo.mjs   기업마당 API(금융/내수/창업/경영) → raw/api·html·files (runId 스냅샷)
  → normalize-notices.py        8필드 정규화
  → build-knowledge-db.py       ★분류: 목적태그14·금액·금리·대상·제외·서류·섹션 + quality 등급
  → build-ml-preallocation.py   중복·변경공고·사업군 dedup
  → build-chatbot-db.py         경량 목록/추천(family 포함)
  → build-chatbot-sections.py   공고 자유추출 섹션 DB
검증: tests/ (회귀) · validate-ml-preallocation.py (재현성 SHA-256)

2-2. 안정화로 추가

  1. 증분 수집: pblancId + updtPnttm로 신규/변경만. 변경 없는 첨부 재다운 생략(852MB 방지)
  2. 신선도: collectedAt·lastChecked·status 저장. dday는 저장 금지, 로드 시 동적 계산(스냅샷 노화 방지)
  3. 스냅샷 버저닝: raw/*/{runId} 불변 보관(최근 N개), 파생 DB는 latest+날짜본
  4. 오케스트레이터 refresh-policyfit.mjs: 수집→정제→분류→빌드→검증 한 방
  5. 검증 게이트: tests/validate 통과해야 latest 교체(실패 시 이전본 유지)
  6. 자동 스케줄: /schedule 스킬(크론) 또는 OS 작업 스케줄러로 일/주 단위

2-3. 정책핏 어댑터

build-policyfit-db.py 신규: knowledge-db.json+ml-preallocation → 정책핏 스키마. match는 진단 슬롯 교차 점수(기존 matchPolicies 이식, 런타임), goals/stages/regions 태그로 선택지별 건수 계산. 산출 policyfit-db.json + 기존 chatbot-sections.json 재사용.

3. 구현 계획 (신규 outputs/policyfit.html)

단일 파일·자체 완결. 기존 policy-chatbot.html의 검증된 함수(매칭·목적축·자유추출 extractFromNotice·이벤트·실패대응)를 이식.

단계내용
P1 디자인 시스템토스 블루 토큰, Pretendard, 카드/칩/버블, 모바일 우선
P2 온보딩5유형 선택 → 유형별 프로필(자동채움/생략/관점)
P3 진단 플로우목적→단계→업종(선택)→지역(선택), 선택지 건수, 1개만 답해도 결과, 건너뛰기
P4 결과매칭도 순 카드(금액·기간·D-day·match%·준비물·steps), 0건 폴백, 면책/1357
P5 부가조건 저장(localStorage), 새 공고 알림(로컬), 전체 둘러보기, "왜 정책핏" 비교
P6 전문가/관리자입안자 브리핑(분야별 문의·마감임박·Top3 + 차트), 관리자 상담 현황 ※문의량은 시연 가상치 명시
P7 자유 추출카드에서 평가기준/우대/제외 토픽 → chatbot-sections.json
