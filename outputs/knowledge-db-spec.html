1. 목적

현재 정규화 데이터(normalized-notices)는 API 메타데이터 + HTML 구조 추출 수준이다. 첨부파일(HWP/PDF → MD)에 들어있는 구체적 지원 내용, 사용 용도, 제한 사항, 신청 절차가 반영되지 않았다.

이 기획서는 795건의 모든 소스(API + HTML + 첨부 MD)를 통합 파싱하여 공고당 구조화된 지식 레코드를 생성하는 스펙이다.

2. 입력 소스

소스경로건수내용
정규화 JSONoutputs/normalized-notices-20260601-224706.json795API 메타 + HTML 구조 필드 8개
상세 HTML MDraw/markdown/{runId}/{noticeId}/detail.md795소관부처, 사업개요, 신청기간, 신청방법, 문의처
첨부파일 MDraw/markdown/{runId}/{noticeId}/*.md (detail.md 제외)1,450본문출력파일, HWP/PDF 변환 텍스트

4. 필드별 추출 규칙

4-1. support.purposes (지원 용도) — 핵심

이것이 가장 중요한 필드. "이 프로그램으로 인테리어 비용을 쓸 수 있나?"에 답하는 필드.
첨부파일에서 "지원항목", "사업비 사용범위", "지원 내용" 등의 섹션을 찾아 구체적 용도를 리스트로 추출.

4-2. support.amount (지원 금액)

패턴: \d+[,\d]*(억원|천만원|백만원|만원) + "이내", "한도", "최대" 앞뒤 문맥

추가: 자부담 비율, 매칭 조건 (예: "투자금의 최대 5배")

4-3. eligibility.exclusions (제외 대상)

첨부파일에서 "지원제외", "참여제한", "제외대상", "다음에 해당하는 자" 섹션을 찾아 리스트로 추출.

4-4. application.steps (신청 절차)

"신청방법", "접수절차", "신청 안내" 섹션에서 순서대로 추출.

4-5. application.documents (필요 서류)

"제출서류", "구비서류" 섹션에서 리스트로 추출.

4-6. quality.grade (정보 품질 등급)

등급기준예상 비율
detailedpurposes 2개+, amount 있음, exclusions 있음~15%
moderatepurposes 1개+, amount 또는 exclusions 있음~55%
vaguepurposes 없음, "공고문 참조" 수준~30%

5. Codex 실행 지침

5-1. 스크립트 위치

scripts/build-knowledge-db.py

5-2. 입력

  • outputs/normalized-notices-20260601-224706.json (기존 정규화 데이터)
  • raw/markdown/20260601-224706/{noticeId}/detail.md (상세 HTML)
  • raw/markdown/20260601-224706/{noticeId}/*.md (첨부 MD)

5-3. 처리 흐름

  1. 정규화 JSON 로드 → 공고 795건 루프
  2. 각 공고에 대해:
    • detail.md 읽기
    • 첨부 *.md 전체 읽기 (합치기)
    • 전체 텍스트에서 섹션별 추출:
      • support.purposes: 용도 태그 키워드 매칭
      • support.amount: 금액 패턴 + 문맥
      • support.rate: 금리/이율 패턴
      • eligibility.exclusions: 제외 섹션 파싱
      • application.steps: 절차 섹션 파싱
      • application.documents: 서류 섹션 파싱
      • contact: 전화번호/이메일 패턴
    • quality.grade 판정
  3. 전체 결과를 outputs/knowledge-db.json으로 저장
  4. 요약 리포트를 outputs/knowledge-db-report.html로 저장

5-4. 핵심 주의사항

  • 첨부 MD는 HWP에서 변환된 것이라 표 구조가 깨져 있을 수 있음. <...> 태그 잔해, 연속 공백 등 정리 필요
  • 같은 정보가 detail.md와 첨부에 중복 등장 가능 → dedup 처리
  • "변경 공고", "수정 공고"는 이전 공고와 동일 제목 → 최신본만 유지 (기존 _hidden 로직 참조)
  • purposes 추출 시 "사용 불가" 맥락 주의: "인건비는 지원 불가"에서 "인건비"를 용도로 잡으면 안 됨

6. 출력 파일

파일설명
outputs/knowledge-db.json795건 전체 지식DB (위 스키마)
outputs/knowledge-db-report.html추출 결과 요약: 등급 분포, 필드 커버리지, 샘플 레코드

7. 검증 기준

  • purposes 추출율: 전체의 50%+ (현재 추정 ~60%)
  • amount 추출율: 전체의 40%+ (현재 39%)
  • exclusions 추출율: 전체의 50%+ (현재 55.7%)
  • quality=detailed: 전체의 10%+
  • "지원 불가" 문맥에서 잘못 추출된 purposes: 0건
