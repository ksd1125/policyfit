정책공고 지식DB 학술 검증형 설계안

2026-06-02 | 대상: 기업마당 정책공고 795건 | 목적: 연구 재현성과 실무 활용성을 함께 확보

권장 결론
현재 규칙 기반 추출기를 폐기하지 않는다. 이를 고정된 정답 생성기가 아니라 후보 생성기 + 약지도 라벨링 함수 집합으로 재정의한다. 문서 구조와 원문 근거를 보존하고, 사람 정답셋과 독립 홀드아웃 평가를 추가한다. 애매한 사례만 제한적으로 모델 또는 LLM이 판정하게 한다.

1. 학술적 유사 문제

학술 문제정책공고 DB와 닮은 점설계에 반영할 점
Template Filling / Information Extraction
MUC 계열		 문서에서 미리 정한 필드에 값을 채운다. 현재의 금액, 금리, 제출서류, 제외조건 추출과 직접 대응한다. 고정 스키마, 독립 테스트셋, 필드별 정밀도·재현율·F1을 기본 평가 틀로 사용한다.
Document-level Event Extraction
Doc2EDAG, DocEE		 필드가 한 문장에 모이지 않고 본문 여러 구역과 첨부 문서에 흩어진다. 한 공고에 여러 지원 프로그램이 함께 들어가기도 한다. 공고 전체를 하나의 레코드로 보되, 하위 프로그램과 근거 문장을 연결한다. 문장 단위 키워드 일치만으로 최종값을 확정하지 않는다.
Knowledge Base Construction from Richly Formatted Data
Fonduer		 HTML, PDF, HWP 변환문, 표 잔해처럼 형식과 스타일의 변이가 크다. 값의 의미가 문서 구조와 표 위치에 좌우된다. 텍스트만 평탄화하지 않고 파일, 섹션, 표, 행, 근거 위치를 보존한다. 구조적 특징을 후보 판정에 사용한다.
Weak Supervision / Data Programming
Snorkel		 키워드, 정규식, 부정 문맥, 섹션 경계는 각각 정확도와 커버리지가 다른 휴리스틱이다. 각 규칙을 독립 라벨링 함수로 기록한다. 규칙 간 충돌과 기권을 보존하고, 사람 라벨로 성능을 추적한다.
Human Annotation Reliability
Krippendorff, Artstein & Poesio		 “지원금액”, “기업별 한도”, “총사업비”, “제외조건”의 경계는 사람도 다르게 판단할 수 있다. 코딩 지침을 만들고 두 명 이상이 독립 주석한다. 합의율과 Krippendorff's alpha를 보고한다.
핵심 해석
이 데이터는 단순한 키워드 분류 문제가 아니다. 다중 형식 문서에서 근거를 가진 슬롯을 채우는 문서 단위 지식베이스 구축 문제에 가깝다.

2. 현재 방식 진단

현재 구성장점한계재정의
용도 키워드 사전빠르고 설명 가능태그 존재 여부는 알지만 근거 위치와 부정·범위 판정이 약함용도 후보 라벨링 함수
금액 정규식 + 최대값 선택숫자 후보 수집에 유효총사업비, 기업별 한도, 세부 프로그램 한도를 혼합할 수 있음금액 후보 생성기. 최종 역할 분류는 별도 수행
금리 문맥 제한명백한 퍼센트 오탐 감소문맥 창만으로는 모든 표현을 포괄하지 못함고정밀 금리 라벨링 함수
섹션 경계 정리표 잔해와 이웃 섹션 혼입 감소평탄화 과정에서 원래 표 구조가 사라짐전처리 규칙 + 문서 구조 보존 계층
quality.grade검토 우선순위 지정에 유용정확도 검증 없이 값 존재 여부로 점수를 계산함reviewPriority로 변경. 학술 지표와 분리
주장 가능한 범위
현재 결과는 “규칙 기반 예비 추출 결과”이다. 사람 정답셋과 홀드아웃 평가 전에는 “정확도 검증 완료” 또는 “학술적으로 검증된 DB”라고 표현하지 않는다.

3. 권장 아키텍처

1. Source Layer API, 상세 HTML, 첨부파일, 변환 MD를 원본 단위로 보존한다.
sourceIdfileType
2. Structure Layer 문서, 섹션, 표, 행, 문장, 문자 위치를 계층화한다.
sectionPathspan
3. Candidate Layer 금액, 금리, 기간, 목적, 제외조건 후보와 원문 근거를 모두 남긴다.
candidateevidence
4. Decision Layer 규칙, 약지도 점수, 제한적 모델 판정으로 후보의 역할과 최종값을 결정한다.
roleconfidence
5. Evaluation Layer 사람 정답셋과 홀드아웃으로 필드별 성능과 오류 유형을 계산한다.
P/R/F1alpha

3-1. 최종 판정 전략

필드군권장 방법이유
전화, 이메일, URL, 등록일 결정적 파서 + 형식 검증 표현 규칙이 비교적 안정적이고 설명 가능성이 높다.
기간, 금액, 금리 후보 생성 + 역할 분류 + 근거 저장 숫자 추출보다 “기업별 한도인지 총사업비인지” 구분이 더 어렵다.
지원 용도 다중 라벨 후보 + 약지도 + 사람 검증 복수 태그가 가능하고 부정 문맥, 세부 프로그램 범위가 영향을 준다.
제외조건, 제출서류, 신청절차 섹션 탐지 + 목록 분할 + 구조 기반 필터 문서 구조가 의미 판정에 직접 관여한다.
모호 사례 근거 구간만 전달하는 제한적 모델 또는 LLM 판정 전체 문서를 자유 생성 방식으로 처리하면 재현성과 추적성이 낮아진다.

3-2. 규칙은 라벨링 함수로 관리

각 규칙은 최종값을 바로 덮어쓰지 않고 label, abstain, evidence, ruleId를 반환한다.

예시 라벨링 함수판정기권 조건
LF_AMOUNT_EXPLICIT_CAP“기업당 최대 5천만원”을 company_cap 후보로 추천주체가 없거나 총사업비 섹션이면 기권
LF_RATE_FINANCE_CONTEXT금리·이율·이차보전 주변의 0~30% 값을 금리 후보로 추천보험료 감면, 자부담 비율이면 기권
LF_PURPOSE_ONLINE_SALES온라인 판로·쇼핑몰 입점·라이브커머스를 온라인판로로 추천온라인 접수만 존재하면 기권
LF_EXCLUSION_SECTION지원제외 섹션 목록을 제외조건 후보로 추천다음 섹션으로 넘어가거나 신호어가 없으면 기권

4. 스키마 개선

기존 챗봇용 레코드는 유지하되, 학술 검증과 오류 분석을 위해 후보와 근거를 추가한다.

{
  "id": "PBLN_...",
  "support": {
    "amount": {
      "selected": {"value": "최대 500만원", "role": "company_cap"},
      "candidates": [
        {
          "value": "최대 500만원",
          "role": "company_cap",
          "sourceId": "attachment-01",
          "sectionPath": ["지원내용", "세부 프로그램"],
          "evidenceText": "업체당 최대 500만원 이내 지원",
          "span": {"start": 1220, "end": 1240},
          "ruleIds": ["LF_AMOUNT_EXPLICIT_CAP"],
          "confidence": 0.96
        }
      ]
    }
  },
  "review": {
    "priority": "high",
    "reasons": ["multiple_amount_roles", "attachment_table_flattened"]
  }
}

quality.grade는 아래처럼 분리한다.

기존변경의미
quality.gradereview.priority사람 검토가 먼저 필요한 레코드의 운영 순위
없음decision.confidence후보 판정기의 신뢰 점수. 보정 여부를 명시
없음evaluation.metrics정답셋과 비교하여 계산한 필드별 검증 지표

5. 검증 프로토콜

5-1. 표본과 분할

795건 전체를 즉시 수작업 라벨링하지 않는다. 먼저 층화표본 240건으로 벤치마크를 만든다.

세트건수용도규칙 수정 허용
Pilot60코딩 지침 작성, 모호 사례 정의, 코더 교육허용
Development100라벨링 함수와 후보 판정기 개선허용
Validation40임계값과 오류 유형 확인제한적 허용
Final Holdout40최종 성능 보고금지

표본은 분야, 첨부 형식, 첨부 수, 본문 길이, 다중 금액 후보 여부, 표 잔해 여부를 기준으로 층화한다.

5-2. 사람 주석

  1. Pilot 60건은 최소 두 명이 독립 주석한다.
  2. 불일치는 adjudicator가 판정하고 지침에 사례를 추가한다.
  3. Development·Validation·Holdout도 이중 주석을 원칙으로 한다. 비용 제약이 있으면 Development 일부만 단일 주석하고 Holdout은 반드시 이중 주석한다.
  4. 각 값에는 원문 근거와 출처 파일을 함께 기록한다.

주석 UI는 지식베이스 채우기와 근거 연결을 지원하는 INCEpTION을 우선 검토한다. 단순 span 주석 중심이면 brat도 사용할 수 있다.

5-3. 평가 지표

대상주요 지표보조 지표
지원 용도 다중 라벨Micro / Macro Precision, Recall, F1태그별 F1, Hamming loss
금액·금리·기간역할 포함 exact match정규화 값 일치율, 후보 recall@k
제외조건·제출서류·절차항목 단위 Precision, Recall, F1근거 span overlap F1
사람 주석 신뢰도Krippendorff's alpha코더 간 pairwise F1, 불일치 유형표
최종 DB필드별 Macro F1문서 단위 완전 일치율, 검토 필요 비율

5-4. 성공 기준

단계권장 게이트
코딩 지침 안정화핵심 필드의 Krippendorff's alpha ≥ 0.80을 목표로 하고, 낮은 필드는 지침과 스키마를 재검토
1차 운영 배포고정밀 필드 Precision ≥ 0.95, 의미 필드 Macro F1 ≥ 0.80, 근거 누락률 ≤ 5%
챗봇 자동 응답고신뢰 판정만 자동 노출. 임계값 미만은 “공고문 확인 필요” 또는 사람 검토 큐로 이동
숫자는 연구 목표에 맞춘 권장 게이트이며 보편 법칙이 아니다. 논문에는 최종 홀드아웃 결과와 신뢰구간, 실패 사례를 함께 보고한다.

6. 실행 로드맵

Phase 0. 코퍼스 동결과 기준선 저장
현재 795건 runId와 추출 결과를 버전 고정한다. 기존 규칙 기반 결과를 baseline으로 남긴다.
Phase 1. 근거 보존형 후보 레이어
금액·금리·기간부터 후보, 역할, 출처 파일, 섹션, 근거 문장을 저장한다. 다중 금액 후보 234건을 우선 오류 분석한다.
Phase 2. Pilot 60건 이중 주석
주석 지침 v0.1을 만들고, 두 코더의 불일치를 분석하여 스키마를 수정한다.
Phase 3. 240건 벤치마크와 라벨링 함수 평가
규칙별 coverage, conflict, abstain, precision을 계산한다. Validation에서 임계값을 고정한다.
Phase 4. 제한적 모델 판정 실험
규칙이 충돌하거나 기권한 사례만 대상으로 구조 기반 분류기 또는 제한적 LLM 판정을 비교한다. 모든 판정은 원문 근거를 반환하게 한다.
Phase 5. Final Holdout 평가와 보고
손대지 않은 40건에서 최종 성능을 계산하고, 재현 가능한 평가 스크립트와 오류표를 함께 보존한다.
