## 5. 지금까지 확인된 핵심 판단

- 이 데이터셋은 `공고 하나 = 문서 하나` 구조가 아님
- 실제 구조는 `공고 하나 = 메타데이터 + 상세 HTML + 본문출력파일 + 첨부 N개`
- 따라서 파싱은 아래 순서가 적절함
  1. 메타데이터 파싱
  2. 상세 HTML 파싱
  3. 본문출력파일 파싱
  4. 첨부파일 파싱

- 챗봇용 정규화도 단일 분류가 아니라 아래 다축 구조가 적절함
  - 문서구조유형
  - 지원유형
  - 대상유형
  - 접수유형
  - 기간유형

- `PDF`, `HWP`, `HWPX`를 `MD` 또는 텍스트로 변환

  - 이미지 기반 파일
  - 압축파일 내부 양식
  - 인코딩 문제

### 6-2. 2단계: 정규화 후보 필드 추출

- 우선 추출 대상
  - 지원대상
  - 지원내용
  - 신청기간
  - 신청방법
  - 제출서류
  - 제외조건
  - 문의처
  - 유의사항

- 이 단계에서는 아직 사용자 표시 범위를 확정하지 않고,
  `안정적으로 뽑히는 항목`과 `첨부를 읽어야만 뽑히는 항목`을 구분

### 6-3. 3단계: 챗봇용 정규화 스키마 확정

- 공고 요약형 응답에 필요한 필드
- 실무형 응답에 필요한 필드
- 출처 추적용 원문 링크와 위치 정보
- 신뢰도 또는 추출 불확실성 플래그

### 6-4. 4단계: 샘플 질의 테스트

- 예:
  - 소상공인 자금 지원
  - 온라인 판매/판로 지원
  - 창업 초기 지원
  - 보증/특례보증

- 목표:
  공고를 단순 나열하는 것이 아니라,
  질문 의도에 맞는 공고 후보와 핵심 항목을 묶어 설명할 수 있는지 점검
