# 기업마당 공고 데이터셋 설계

작성일: 2026-06-01

## 1. 목적

이 프로젝트의 목적은 기업마당 API를 이용해 최근 2년간 `창업`, `경영`, `금융`, `내수` 분야 공고를 전량 수집하고, 상세페이지와 첨부파일까지 원문을 확보한 뒤 MD로 변환하여 정규화 가능한 데이터 재료를 파악하는 것이다.

지금 단계의 목표는 챗봇 답변 화면을 만드는 것이 아니라 다음 질문에 답할 수 있는 데이터셋을 만드는 것이다.

- 최근 2년 공고 중 실제로 수집 가능한 공고는 몇 건인가
- 공고별로 확보 가능한 원문은 무엇인가
- HTML, PDF, HWP를 MD로 안정적으로 변환할 수 있는가
- 공고에서 반복적으로 등장하는 항목은 무엇인가
- 사용자에게 나중에 어디까지 구조화해 보여줄 수 있는가

## 2. 범위

- API 키나 민감정보의 파일 저장

## 3. 기본 방향

### 3.1 수집과 해석을 분리한다

먼저 최대한 많은 원문을 안전하게 모으고, 그다음 무엇을 정규화할 수 있는지 판단한다. 메타데이터 수집 단계에서 정규화 항목을 미리 단정하지 않는다.

### 3.2 공고 하나를 여러 원문 단위로 보관한다

공고 하나에 대해 아래 자료를 각각 별도로 저장한다.

- API 메타데이터
- 상세페이지 HTML
- 본문출력파일
- 첨부파일 원본
- 원문별 MD 변환본

이렇게 해야 나중에 어떤 정보가 어디에서 왔는지 추적할 수 있다.

### 3.3 실패도 데이터로 남긴다

수집 실패, 다운로드 실패, 변환 실패, 인코딩 깨짐, 중복 공고 여부를 모두 로그로 남긴다. 그래야 실제 학습 가능한 재료의 범위를 판단할 수 있다.

## 5. 수집 파이프라인

### 5.1 1차 수집: API 메타데이터 전량 수집

기업마당 API를 분야별로 반복 호출하여 최근 2년 공고 메타데이터를 전량 확보한다.

이 단계에서 우선 저장할 메타데이터:

- 공고 ID
- 공고명
- 공고 URL
- 소관기관명
- 수행기관명
- 지원분야
- 해시태그
- 등록일자
- 신청기간
- 지원대상
- 사업개요 요약
- 첨부파일 URL
- 본문출력파일 URL

### 5.2 2차 수집: 공고 원문 확보

각 공고마다 확보 가능한 원문을 가능한 한 모두 가져온다.

- 상세페이지 HTML
- 본문출력파일
- 첨부 PDF/HWP 파일

공고별 원문 확보 상태를 별도 매니페스트로 남긴다.

### 5.3 3차 처리: MD 변환

원문을 아래 방식으로 변환한다.

- HTML -> MD
- PDF -> MD
- HWP -> 가능한 범위 내 텍스트/MD

원문을 강제로 하나로 합치지 않고 출처별로 따로 저장한다.

예시:

- `notice.md`
- `print_file.md`
- `attachment_01.md`
- `attachment_02.md`

### 5.4 4차 분석: 재료 지도 작성

MD 변환 결과를 바탕으로 다음을 확인한다.

- 어떤 필드가 반복적으로 등장하는가
- 어떤 필드는 상세페이지에서만 확보되는가
- 어떤 필드는 첨부파일을 읽어야만 확보되는가
- 어떤 문서 유형에서 변환 실패가 많이 생기는가

## 6. 분석 관점

초기 분석 대상 항목은 아래 정도로 본다.

- 지원대상

- 지원내용
- 신청방법

- 항상 안정적으로 추출되는 항목
- 가끔만 추출되는 항목
- 첨부파일을 읽어야만 보이는 항목

## 9. 보안과 운영 원칙

- API 키는 코드나 문서에 저장하지 않는다
- 민감정보가 포함된 문서는 저장하지 않는다
- 원문이 공개자료인지 확인한 후만 저장한다
- 대용량 파일은 코드 저장소와 분리된 본 프로젝트 폴더에만 둔다
